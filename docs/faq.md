# FAQ

## 概要

本書は People Success Navigator の現状実装を前提に、引き継ぎ時や日常運用で発生しやすい質問をまとめた FAQ です。  
厳密な設計原本というより、**開発者・運用担当者が最初に困りやすい点を素早く確認するための実務向けメモ** として整理しています。

関連資料:

- `README.md`: セットアップ・起動方法
- `api_spec.md`: API 一覧
- `architecture.md`: システム構成
- `db_design.md`: DB 設計
- `screen_spec.md`: 画面仕様
- `screen_flow.md`: 画面遷移
- `usecase.md`: ユースケース
- `operation_manual.md`: 運用手順

## 1. セットアップ・起動まわり

### Q1. このシステムは何で構成されていますか？

A. 現状は以下の構成です。

- フロントエンド: 静的 HTML / CSS / JavaScript
- バックエンド: FastAPI
- DB: SQLite（ローカル開発時）
- 外部連携: Dify API

ローカルでは、バックエンドを `uvicorn app.main:app --reload` で起動し、フロントは Live Server 等で `backend/app/static/` 配下の画面を開く想定です。

### Q2. 最低限どこを起動すれば動作確認できますか？

A. 最低限、以下を確認できれば基本確認は可能です。

1. バックエンドを起動する
2. `http://127.0.0.1:8000/docs` が開けることを確認する
3. フロントの `login.html` を開く
4. ログイン後、ロール別ダッシュボードへ遷移できることを確認する

### Q3. フロントは FastAPI 配下で配信していますか？

A. 現状の開発前提では、FastAPI と静的フロントを分離して確認する形です。  
フロント資産は `backend/app/static/` にありますが、README では Live Server 等での起動を想定しています。

### Q4. DB は何を使っていますか？

A. ローカル開発では SQLite を使っています。既定では `backend/app.db` が対象です。  
将来的に RDBMS を差し替える余地はありますが、現状資料は SQLite 前提です。

### Q5. Python や依存ライブラリの厳密な推奨バージョンはありますか？

A. 現状資料では厳密固定にはしていません。  
開発途中のため、README では「現行コードが動く範囲で整える」前提に留めています。バージョン固定を行う場合は、今後 `requirements.txt` やロックファイルに合わせて明文化する想定です。

## 2. 環境変数・設定

### Q6. `.env` は必須ですか？

A. 完全に必須ではありませんが、実運用や Dify 連携確認をする場合は設定した方がよいです。  
特に `DATABASE_URL`、JWT 関連設定、Dify API Key は確認対象になります。

### Q7. `.env.example` はどれを見ればいいですか？

A. ルートに雛形を置く運用にしています。  
実際にどちらを使うかは起動方法や読み込み実装に依存するため、現状は「開発時に参照しやすい雛形」として用意しています。

### Q8. Dify の設定がなくても起動できますか？

A. 画面表示や基本 API の確認はできる可能性がありますが、AI 分析系の処理は期待通り動かない可能性があります。  
特に Pulse のメモ分析や AI 相談系の今後拡張では Dify 設定が影響します。

### Q9. 既存の `.env` をそのまま GitHub に上げてよいですか？

A. 推奨しません。  
API Key や秘密情報が含まれる可能性があるため、GitHub には `.env.example` のみを置き、実値は別管理にしてください。

## 3. 認証・権限

### Q10. ログイン後の遷移先はどう決まりますか？

A. `POST /auth/login` でトークン取得後、`GET /users/me` でロールを取得し、以下へ振り分けます。

- `user` → `dashboard-user.html`
- `manager` → `dashboard-manager.html`
- `admin` → `dashboard-admin.html`

### Q11. トークンはどこに保存されていますか？

A. フロント側では `localStorage.access_token` に保存する実装前提です。

### Q12. 未ログインや権限不一致のときはどうなりますか？

A. 画面側の共通制御で `GET /users/me` により認証状態とロールを確認し、必要に応じてログイン画面または許可されたダッシュボードへリダイレクトします。

### Q13. ロールは何種類ありますか？

A. 現状は 3 種です。

- `user`
- `manager`
- `admin`

### Q14. manager は何でも見られますか？

A. いいえ。原則として自分配下のメンバーに関する情報が中心です。  
全ユーザー・全部署を前提に扱う操作は `admin` 向けです。

- `skills.html`
- `ai-chat.html`

### Q17. `member-detail.html` はメニューから直接開く画面ですか？

戻り先は `from` パラメータ等の導線前提で制御される想定です。

### Q18. `dashboard.html` は何のためにありますか？

A. 業務画面というより、認証済みユーザーをロール別ダッシュボードへ振り分ける中継ページです。

### Q19. `index.html` は使っていますか？

A. 現状は未使用または空ファイルとして扱っています。正式なエントリーポイントとしては想定していません。

## 5. Pulse・分析・離職リスク

### Q20. Pulse は何を保存していますか？

A. 主に以下です。

- 回答者ユーザーID
- コンディションスコア
- 自由記述メモ
- 対象日
- 登録日時

`user_id + survey_date` の一意制約があるため、同一ユーザーの同日重複登録は制限されます。

### Q21. Pulse 登録時に AI 分析は入りますか？

A. 現状資料では、メモが入力されている場合に AI 分析を実施し、必要に応じてスコア補正がかかる実装を前提に整理しています。

### Q22. 離職リスク判定はいつ実行されますか？

A. 現状では手動実行と日次運用の両方を想定しています。  
マネージャー画面からの手動再判定に加え、将来的または運用補助として「毎日決まった時刻に前日分までを判定する」形が相性の良い運用です。

### Q23. アラートは誰が確認・解消しますか？

A. 主には `manager` が確認し、必要に応じて解消します。  
`team-overview.html` や `dashboard-manager.html` の導線が中心です。

### Q24. リスク判定ロジックはどこにありますか？

A. 資料上は `risk_judge_service.py` を中心にしたサービス層へ寄せる整理が自然です。  
ただし現状コードでは一部が endpoint 側に寄っている可能性があるため、今後のリファクタ対象として見ています。

## 6. ユーザー・部署管理

### Q25. ユーザー作成は誰ができますか？

A. 現状は `admin` のみです。  
API では `POST /users/` を利用します。

### Q26. 管理画面ではどの情報を変更できますか？

A. 現状資料では、主に以下を変更対象にしています。

- ロール
- 所属部署
- 上司（manager）
- ユーザーの追加・削除

### Q27. 部署を削除できないことがあるのはなぜですか？

A. その部署に所属ユーザーが残っている場合、整合性維持のため API 上削除不可としている前提です。

### Q28. ユーザー削除時、配下メンバーの manager_id はどうなりますか？

A. 現状資料では `null` に更新される実装前提で整理しています。

### Q29. 初期管理者はどうやって作りますか？

A. 初期セットアップ用スクリプトを実行する想定です。  
例として `python scripts/create_initial_admin.py` を利用します。

### Q30. API のベース URL はどこですか？

A. 現状資料では `http://127.0.0.1:8000/api/v1` を前提にしています。

### Q31. API の確認はどこでできますか？

A. FastAPI の Swagger UI を使います。  
`http://127.0.0.1:8000/docs` で確認できます。

### Q32. DB 設計はコードと完全一致していますか？

開発途中のため、将来変更の可能性があります。

### Q33. このシステムは service 層に全部まとまっていますか？

A. まだ完全ではありません。  
CRUD / service / endpoint の分離方針はありますが、開発途中のため endpoint にロジックが残っている箇所も想定しています。

### Q34. 本番 DB や本番構成まで資料化されていますか？

A. まだ詳細固定していません。  
今回の資料群はローカル開発・検証前提の情報を中心にしています。

## 8. トラブルシュート

### Q35. ログインできないときは何を確認すればよいですか？

A. まず以下を確認します。

1. バックエンドが起動しているか
2. `/docs` が開くか
3. 初期ユーザーや対象アカウントが存在するか
4. メールアドレス・パスワードが正しいか
5. ブラウザコンソールに CORS や 401 エラーが出ていないか

### Q36. 画面が真っ白、またはデータが出ないときは？

A. 以下の切り分けが有効です。

- ブラウザコンソールに JavaScript エラーがないか
- Network タブで API が失敗していないか
- `localStorage.access_token` が残っているか
- API のレスポンス形式変更がフロントに未反映ではないか

### Q37. `team-overview.html` の絞り込みや並び替えが効かないときは？

A. フロント側の状態更新と再描画処理の不整合が起きやすい箇所です。  
確認ポイントは以下です。

- 絞り込み条件の state が更新されているか
- ソート後に描画関数を再実行しているか
- API 取得データと表示対象配列を分けて管理しているか
- DOM 更新前後でイベント再設定が必要になっていないか

### Q38. Dify 連携が動かないときは？

A. まず環境変数を確認します。

- API Key が設定されているか
- エンドポイント URL が正しいか
- ネットワーク到達性に問題がないか
- Dify 側アプリ設定が変更されていないか

### Q39. DB を壊した・状態を戻したいときは？

A. ローカル検証なら、バックアップを戻すか、必要に応じて DB を再作成して初期投入スクリプトを流し直す運用が現実的です。  
SQLite 単体ファイル運用のため、復旧は比較的シンプルです。

## 10. 更新ルールの目安

### Q43. FAQ はどんなときに更新すべきですか？

A. 少なくとも以下のタイミングで更新すると有効です。

- 起動手順が変わったとき
- API パスや認証方式が変わったとき
- ロール権限が変わったとき
- 実運用で同じ質問が複数回出たとき
- モック画面が正式実装になったとき

### Q44. FAQ の粒度はどのくらいがよいですか？

A. 設計書の代替にしすぎず、**「初見の担当者が困ったときに最初に見る資料」** として保つのがよいです。  
詳細仕様は各設計書へ分け、FAQ には結論と参照先を載せる運用が扱いやすいです。
