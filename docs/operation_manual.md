# 運用手順書

## 概要

本書は People Success Navigator の**現状実装に基づく運用手順**を整理した資料です。  
設計思想や画面一覧ではなく、日次運用・定期確認・データ整備・トラブル時の一次対応など、実際の運用担当者が行う作業を中心にまとめています。

前提:

- 本システムは FastAPI バックエンド + 静的 HTML / JavaScript フロントで構成されています。
- ローカル開発・検証では `backend/app.db` の SQLite を利用します。
- ロールは `user` / `manager` / `admin` の 3 種です。
- 一部画面はモック段階のため、運用対象は主に認証・Pulse・チーム確認・社員管理・離職リスク判定です。

関連資料:

- `README.md`: セットアップ・起動方法
- `api_spec.md`: API 一覧
- `architecture.md`: システム構成
- `db_design.md`: DB 設計
- `screen_spec.md`: 画面別仕様
- `screen_flow.md`: 画面遷移
- `usecase.md`: 業務フロー別ユースケース

## 2. 運用対象機能

現状で実運用または検証運用の対象として扱いやすい機能は以下です。

| 機能 | 主担当 | 概要 |
|---|---|---|
| ログイン | 全ロール | 認証とロール別振り分け |

| チーム状況確認 | manager / admin | 一覧、絞り込み、並び替え、詳細確認 |

## 3. 運用開始前の確認

## 3.1 アプリケーション起動確認

### 目的

- API と画面が最低限利用可能な状態か確認する

### 手順

1. `backend/` へ移動する
2. 必要に応じて仮想環境を有効化する
3. `uvicorn app.main:app --reload` を実行する
4. ブラウザで `http://127.0.0.1:8000/docs` を開く
5. フロント画面を Live Server 等で表示する
6. `login.html` が表示できることを確認する

### 確認観点

- FastAPI が起動している
- Swagger UI が表示される
- フロントから API へアクセスできる
- CORS エラーが出ていない

## 3.2 初期データ確認

### 確認対象

- 初期管理者アカウントが存在する
- 必要な部署が存在する
- チーム確認画面用のサンプルデータが投入済み、または実データが存在する

### 利用スクリプト例

```bash
python scripts/create_initial_admin.py
python scripts/create_initial_departments.py
python scripts/seed_team_overview_data.py
```

## 3.3 環境変数確認

最低限、以下を確認します。

| 変数名 | 確認内容 |
|---|---|
| `DATABASE_URL` | 接続先が正しいか |
| `PROJECT_NAME` | 任意だが起動時設定として妥当か |
| `DIFY_API_KEY` | 利用する場合のみ設定されているか |
| `DIFY_API_KEY2` | 予備・別用途があれば設定されているか |

## 4. 日次運用

## 4.1 利用者の Pulse 入力

### 主担当

- `user`
- `manager`
- `admin`

### 目的

- 毎日のコンディションデータを蓄積する

### 基本手順

1. 利用者がログインする
2. `pulse.html` を開く
3. 当日分の Pulse を入力する
4. 送信後、履歴や最新表示が更新されることを確認する

### 運用上の確認ポイント

- 当日未入力者が増えていないか
- 直近スコアが大きく下がっている利用者がいないか
- 継続入力率が維持できているか

## 4.2 マネージャーによるチーム確認

### 主担当

- `manager`

### 目的

- 当日または直近のチーム状態を把握し、フォロー対象を見つける

### 基本手順

1. `dashboard-manager.html` を開く
2. チーム人数、平均スコア、リスク分布、アラート件数を確認する
3. 必要に応じて `team-overview.html` へ遷移する
4. 絞り込み・並び替えを利用し、高リスクまたは未確認メンバーを抽出する
5. 詳細確認が必要な場合は `member-detail.html` を開く

### 運用判断の例

- 離職リスクが `high` のメンバーを優先確認する
- Pulse 未入力が続くメンバーをフォロー候補とする
- アラート未解消件数が多い場合は優先的に処理する

## 4.3 アラート確認と解消

### 主担当

- `manager`

### 目的

- 要対応アラートを見落とさず、対処済みのものを整理する

### 基本手順

1. `dashboard-manager.html` でアラート一覧を確認する
2. 対象メンバーの詳細を確認する
3. 必要に応じて 1on1、声掛け、状況確認などの実業務対応を行う
4. 対応完了後、アラート解消操作を実施する
5. 一覧から消えたこと、または解消状態に変わったことを確認する

### 運用メモ

- システム上の「解消」は、実業務上のフォロー完了とは必ずしも同義ではありません。
- 運用ルール上、誰が・いつ・どの対応をしたかは別途記録運用を定めると安定します。

## 5. 定期運用

## 5.1 離職リスク日次判定

### 主担当

- 開発/保守担当
- 必要に応じて manager

### 目的

- 前日までの Pulse や判定材料をもとに、リスク状態を更新する

### 実行方法

```bash
python scripts/run_daily_risk_judge.py
```

オプション例:

```bash
python scripts/run_daily_risk_judge.py --days 7 --target-date 2026-03-17
```

### 推奨運用イメージ

- 毎日 1 回、朝の確認前に実行する
- `--target-date` 未指定時は前日基準のため、日次バッチとして扱いやすい
- manager が手動再判定を行う場合は、運用ルール上「いつ再実行してよいか」を決めておく

### 実行後確認

- エラーなく終了したか
- 対象ユーザーのリスク状態が更新されているか
- チーム一覧やダッシュボードの件数表示に反映されているか

## 5.2 新規利用開始時の初期設定

### 主担当

- `admin`

### 目的

- 新入社員や新規利用者を利用可能状態にする

### 基本手順

1. 必要に応じて部署を作成する
2. `employee-management.html` を開く
3. 社員を新規登録する
4. ロールを設定する
5. 所属部署を設定する
6. manager 配下であれば上司を設定する
7. 初回ログイン可能であることを確認する

### 確認項目

- メールアドレス重複がない
- `role` が想定通りである
- `department_id` と `manager_id` の整合が取れている

## 5.3 組織変更時のメンテナンス

### 主担当

- `admin`

### 目的

- 異動・昇格・組織変更をシステムに反映する

### 対応例

- ユーザーのロール変更
- 所属部署変更
- 上司変更
- 不要ユーザーの削除
- 不要部署の整理

### 注意点

- manager を削除した場合、配下ユーザーの `manager_id` は `null` になる実装です。
- 所属ユーザーが残っている部署は削除不可です。
- 人事上の変更タイミングとシステム反映タイミングをずらす場合は、運用ルールを別途決める必要があります。

## 6. 画面別運用手順

## 6.1 `login.html`

### 利用目的

- 認証を行い、ロールに応じたホーム画面へ入る

### 主な操作

1. メールアドレス入力
2. パスワード入力
3. ログイン実行

### 正常系確認

- `access_token` が保存される
- `GET /users/me` でロール取得できる
- ロール別ダッシュボードへ遷移する

## 6.2 `pulse.html`

### 利用目的

- 当日分 Pulse の入力と履歴確認

### 主な操作

1. 当日分の入力
2. 保存
3. 履歴の確認

### 確認ポイント

- 保存直後に一覧や統計へ反映されるか
- 既存入力日の重複扱いが想定通りか

## 6.3 `dashboard-manager.html`

### 利用目的

- チーム運用の起点画面

### 主な確認項目

- チームサマリ
- リスク分布
- アラート一覧
- 要確認メンバー

### よく行う操作

- 高リスク対象の確認
- メンバー詳細画面への遷移
- 手動の離職リスク再判定
- アラート解消

## 6.4 `team-overview.html`

### 利用目的

- 一覧ベースでチーム状態を確認する

### 主な操作

1. 絞り込み条件を選ぶ
2. 並び替え条件を選ぶ
3. 一覧から対象メンバーを選ぶ
4. `member-detail.html` へ遷移する

### 確認ポイント

- 絞り込み条件が一覧に反映される
- 並び替え結果が期待通りである
- 離職リスク順など運用上見たい順序で確認できる

## 6.5 `member-detail.html`

### 利用目的

- 個別メンバーの状態を深く確認する

### 主な確認項目

- 基本プロフィール
- Pulse 履歴
- スコア推移
- 直近の気になる変化

### 運用メモ

- 戻り先は `from` パラメータに依存する実装のため、どの画面から来たかで戻り先が変わります。
- 一覧確認中の導線を壊さないよう、通常はダッシュボードまたは一覧から開く運用が扱いやすいです。

## 6.6 `employee-management.html`

### 利用目的

- 社員情報の登録・更新・削除

### 主な操作

1. 一覧確認
2. 社員新規作成
3. ロール変更
4. 所属部署変更
5. 上司変更
6. 削除
7. 必要に応じて詳細画面確認

### 確認ポイント

- 更新内容が再読込後も保持されている
- 削除後に一覧から消えている
- manager 配下関係が意図通りに維持されている

## 7. 障害・問い合わせ時の一次切り分け

## 7.1 ログインできない

### 確認順序

1. メールアドレス/パスワード誤りがないか
2. 初期管理者アカウントが存在するか
3. API が起動しているか
4. `/auth/login` が 401 / 500 を返していないか
5. ブラウザの開発者ツールでネットワークエラーが出ていないか

### 想定原因

- 認証情報誤り
- DB データ未投入
- API 未起動
- CORS 設定不一致
- トークン保存失敗

## 7.2 画面は開くがデータが出ない

### 確認順序

1. API サーバーが起動しているか
2. Bearer Token が送信されているか
3. 対象データが DB に存在するか
4. ロール権限で閲覧対象外になっていないか
5. JavaScript コンソールにエラーが出ていないか

### 想定原因

- トークン期限切れ
- API 取得失敗
- 初期データ不足
- 権限制御による取得不可
- フロント側スクリプト不整合

## 7.3 チーム一覧の件数や状態がおかしい

### 確認順序

1. リスク判定バッチを最近実行したか
2. Pulse データが入力されているか
3. 絞り込み条件・並び替え条件が残っていないか
4. manager 配下設定が正しいか

### 想定原因

- 判定未実行
- 元データ不足
- フィルタ条件の残留
- manager 配下関係不整合

## 7.4 ユーザー削除や部署削除ができない

### 確認順序

1. admin 権限で実行しているか
2. 削除対象に依存データが残っていないか
3. manager 削除時の配下影響を理解しているか
4. 部署に所属ユーザーが残っていないか

## 8. データメンテナンス方針

## 8.1 ユーザー情報

運用上、以下の整合性を保つことが重要です。

- `email` は一意
- `role` は `user / manager / admin` のいずれか
- `manager_id` は自己参照にならない
- `department_id` は存在する部署を参照する

## 8.2 Pulse データ

運用上の観点:

- 日次蓄積が分析・判定の前提になる
- 未入力者が増えると判定精度や一覧の有用性が下がる
- テストデータと実データを混在させる場合は区分ルールを決めるとよい

## 8.3 アラート/リスク情報

運用上の観点:

- システム上の自動判定結果であり、人事判断そのものではない
- manager の一次確認のきっかけとして利用する
- 解消運用の定義を明確にしないと件数の意味がぶれやすい

## 9. 定例運用チェックリスト

## 9.1 日次チェック

- API が起動している
- ログインできる
- Pulse 入力が可能
- マネージャーダッシュボードが表示できる
- アラート一覧が取得できる
- 必要に応じて離職リスク判定を実行済み

## 9.2 週次チェック

- 新規ユーザー・異動・退職反映漏れがない
- manager 配下設定に不整合がない
- 部署情報に不要データがない
- テストデータが残りすぎていない
- 高リスク対象の未対応放置がない

## 9.3 リリース・改修時チェック

- 主要画面が開く
- ロール別リダイレクトが正しい
- 社員管理操作が正常
- リスク判定バッチが動く
- 既存データとの互換性に問題がない

## 10. 既知の前提・注意事項

- 本システムは開発途中の成果物であり、画面によって実装度合いが異なります。
- 一部集計や表示ロジックはフロント側 JavaScript に依存しています。
- 本番運用を前提にした監視、バックアップ、権限申請フロー、監査ログなどは本資料では未整理です。
- SQLite ベースのため、同時更新や本格運用時の制約は別途考慮が必要です。
- Dify 連携は環境変数設定や接続先前提に依存するため、検証環境ごとに確認が必要です。
